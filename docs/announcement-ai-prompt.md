# 公告產生器 — Gemini 指令（Apps Script 的 `PROMPT`）

公告流程：原始公告（PDF／圖片，來自 iOS 捷徑或表單）→ **Google Apps Script 呼叫 Gemini API**
（`gemini-2.5-flash`，`responseMimeType: application/json`）→ Gemini 回傳 `{date, source, summary}`
→ Apps Script 的 `insertTop_()` 寫進試算表 `公告` 分頁（A 時戳 / B 日期 / C 來源 / D 摘要）
→ `notices.html` 讀取顯示（摘要第 1 行＝標題、其餘＝內文）。

要優化輸出品質，只需替換 Apps Script 裡的 `const PROMPT`。JSON 欄位（`date`/`source`/`summary`）
與程式其餘部分維持不變。

---

## 優化後的 PROMPT（可直接整段取代原本的 `const PROMPT`）

```javascript
const PROMPT = [
  '你是星宇航空(STARLUX)的飛航公告整理助手。請閱讀這份公告（PDF 或圖片），抽取並改寫成 JSON。',
  '只回傳 JSON，不要任何其他文字或 markdown。欄位：date、source、summary。',
  '',
  '【date】發布日期，格式 YYYY/M/D（例：2026/7/10）；找不到就用收到的日期。不要用「2026年7月10日」等其他寫法。',
  '',
  '【source】來源。若是 Fleet Notice 就填其編號（例：FN-26-0060）；若來自 Outlook／Teams／Line，就填該管道名稱。',
  '',
  '【summary】一段文字：第一行是標題，第二行起是內文，內文各重點之間用換行分隔。',
  '  標題：不超過 50 字的中文，點出「主題＋關鍵結果＋日期」。',
  '  內文請依下列順序組織，每個重點自成一行、只講一件事，若原文沒有某一項就略過該行、不要硬湊：',
  '   1. 結論先行：第一行就講最關鍵的結果（是否核准／生效或簽收日期／具體條件或數字）。',
  '   2. 法規／文件狀態：相關 FOM、Revision、Ops Specs（運行規範）的修訂或納入狀態。',
  '   3. 目前限制：現在還「不允許」做什麼、限制到什麼時間點或什麼條件成立為止。',
  '   4. 適用範圍：適用哪些機隊、有哪些例外，以及後續會如何補齊。',
  '   5. 後續與細節：最終／正式授權由誰、在什麼條件下核發；更多細節何時公布。',
  '  若本公告要求組員採取具體行動（下載安裝、於期限前簽收、回報等），',
  '  請在 summary 的「最後一行」加上「需要動作：」開頭的一句話，簡短寫出組員要做什麼',
  '  （例：需要動作：下載並安裝最新版 App）。若不需任何行動就不要加這行。',
  '',
  '【寫作規則】',
  '- 保留原文所有具體數字與條件（例：90 天內完成 100 次成功進場、簽收期限），一律不可省略。',
  '- 專有縮寫第一次出現時附全名，例：民航局（CAA）、RNP AR（Authorization Required）。',
  '- 一行只講一個主題，不要把多件事塞進同一行。',
  '- 用語沿用公司慣用詞：運行驗證／運行授權／運行規範（Ops Specs），不要改成「操作」。',
  '- 重點直接寫內容本身，不要加「需執行事項：」「重點：」之類的前綴標籤。',
  '- 只依原文改寫，不要杜撰原文沒有的事實、日期或數字。'
].join('\n');
```

## 搭配的 notices.html 呈現

`notices.html` 已同步更新，讓公告卡片展開後：

- 內文每一行渲染成**項目符號（bullet）**清單。
- `需要動作：…` 那一行會被抽出，顯示為底部的 **⚡ 需要動作** 橘色標章。
- 含「需要動作」的公告，卡片會自動帶**橘色左邊框**（原本 `.card.important` 樣式先前未套用，一併修好）。

因此 Gemini 只要照上面規則把動作寫進 summary 最後一行，前端就會自動排版，不需再改欄位或 GAS。

## 跟原本 PROMPT 的差異

| 項目 | 原本 | 改後 |
|---|---|---|
| 內文結構 | 「每行一個重點」，無順序 | 強制五段順序：結論 → 法規 → 限制 → 適用 → 後續 |
| 具體數字 | 未特別要求 | 明文禁止省略（90 天／100 次、簽收期限等） |
| 縮寫 | 無 | 首次出現附全名（CAA、RNP AR…） |
| 日期 | YYYY/M/D | 一樣，但明確禁止「〇年〇月〇日」 |
| 用語 | 無 | 統一「運行」（若 FOM 正式用「操作」可自行改） |

> `date` / `source` / `summary` 的 JSON 欄位名、`analyze_()` 中
> `Array.isArray(o.summary) ? o.summary.join('\n') : ...` 的處理、`insertTop_()` 全部無需更動。

## 範例輸出（RNP AR 臨時授權）

```
RNP AR 運行授權申請已獲 CAA 核准，2026/7/10 發布臨時授權
星宇航空 RNP AR（Authorization Required）運行授權申請已獲民航局（CAA）核准。CAA 將於 2026/7/10 發布「臨時授權」，允許星宇航空在 90 天內完成 100 次成功的 RNP AR 進場，以進行運行驗證。
FOM 第 7.10.1.B 節已依 Revision 18-TR01 修訂，納入相關 RNP AR 條款，作為運行驗證的準備措施。
儘管相關 FOM 條款已納入，在授權正式納入星宇航空運行規範（Ops Specs）並宣布運行驗證開始前，仍不允許執行 RNP AR 運行。
本次 RNP AR 運行授權申請僅適用於 A321 與 A330 機隊，A350 機隊不在此次申請範圍內；FOM 將於後續修訂中更新，以明確反映此適用範圍。
正式（最終，非臨時）RNP AR 運行授權，將於星宇航空在臨時授權下成功完成運行驗證後，由 CAA 核發。運行驗證的進一步細節將於適當時候公布。
```

## 為什麼比舊版清楚

- **結論先行**：先講「已核准 + 發布日期 + 驗證條件」，而不是從 FOM 修訂講起。
- **具體條件**：保留「90 天 / 100 次成功進場」的驗證門檻。
- **一行一主題**：法規狀態、目前限制、適用機隊、最終授權各自成行。
- **日期格式固定**：YYYY/M/D 斜線，與公告欄其他資料一致。
